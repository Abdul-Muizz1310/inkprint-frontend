# 02 — E2E (Playwright)

## Goal

One Playwright test in `tests/e2e/cert-flow.spec.ts` that exercises the **happy-path fingerprint flow**, end to end, against the **live backend** at `https://inkprint-backend.onrender.com`.

The single test the PLAN demands ([PLAN.md §9.3 3b](../../../docs/PLAN.md)):

> Playwright: paste text → issue cert → see styled card

Everything else in this file is the supporting cast — config, fixtures, non-blocking smoke checks.

## Scope

### In scope (blocking for S4 gate)
- **`cert-flow.spec.ts`** — the happy-path test above.

### Out of scope for S4 (can be added in S6 polish)
- Verify page flow
- Compare page flow
- Leak scan streaming flow
- Visual regression / screenshot diffing
- Cross-browser (only chromium for S4)

---

## Environment

- **baseURL**: `process.env.NEXT_PUBLIC_SITE_URL ?? "http://localhost:3000"` (set in `playwright.config.ts`).
- **backend**: real, live, at `NEXT_PUBLIC_API_URL=https://inkprint-backend.onrender.com`.
- **No mocking** for the blocking test — we want a real end-to-end signal. Backend is always-warm per its Render config, so cold-start is not a concern.

### Why real backend, not mocks
1. Proves the deployed URL is wired.
2. Catches CORS regressions immediately.
3. Catches schema drift between frontend Zod parsers and backend responses.
4. The backend is public and cheap; there's no authorization to worry about.

### Fixtures
- A hard-coded plain-text string, ~200 chars, deterministic (not random), used by the fingerprint test. Committed as a constant inside the spec file.
- A hard-coded `author` = `"e2e@inkprint.local"`.

### Cleanup
- None. Certificates are immutable records; leaving behind a test certificate per CI run is acceptable. If this ever becomes noisy, add a `[E2E]` prefix in the author field and filter in the backend.

## The spec — `tests/e2e/cert-flow.spec.ts`

### Flow
1. Navigate to `/`.
2. Assert the page renders with the editor, the author input, the fingerprint button, and the legal disclaimer.
3. Paste the fixture text into the Monaco editor.
4. Fill the author field.
5. Click **Fingerprint**.
6. Wait for navigation to `/certificates/<uuid>`.
7. Assert the certificate page renders:
   - Serif headline "Certificate of Authorship"
   - The author value appears
   - The hash row is present and contains a truncated value
   - The QR code image is present
   - The digest preview box is present
   - The verify footer is present and contains the cert id
   - The "Download manifest" and "Share" buttons are present

### Test cases enumerated

| # | Name | Gate |
|---|---|---|
| 1 | `landing page renders editor + author + disclaimer` | blocking |
| 2 | `happy path: paste + fingerprint → certificate page renders` | blocking |
| 3 | `certificate page shows all 4 metadata rows` | blocking |
| 4 | `certificate page shows QR code` | blocking |
| 5 | `certificate page shows verify footer with cert id` | blocking |

The five cases will be expressed as a single Playwright `test.describe("fingerprint happy path", ...)` block with one or more ordered `test(...)` steps. Ordering matters because 2–5 share navigation state from 1; use `test.describe.configure({ mode: "serial" })` or a single `test(...)` with multiple assertions. Prefer the single-test form for speed in CI.

### Assertions
- Navigation assertion: `await expect(page).toHaveURL(/\/certificates\/[0-9a-f-]{36}$/)`
- Headline: `await expect(page.getByRole("heading", { name: /certificate of authorship/i })).toBeVisible()`
- Author row: `await expect(page.getByText(/e2e@inkprint.local/)).toBeVisible()`
- Hash row: `await expect(page.getByTestId("cert-hash")).toContainText("…")`  *(we'll add a `data-testid` on the hash value)*
- QR image: `await expect(page.getByTestId("cert-qr")).toBeVisible()`
- Verify footer: `await expect(page.getByText(/verify at/i)).toBeVisible()`

### Test IDs to add in S4 (so E2E is robust against copy changes)
- `data-testid="editor-root"` on the Monaco editor container (or fallback textarea)
- `data-testid="editor-author-input"` on the author `<input>`
- `data-testid="editor-fingerprint-button"` on the submit button
- `data-testid="legal-disclaimer"` on the disclaimer card
- `data-testid="cert-headline"` on the certificate headline
- `data-testid="cert-author"` on the author metadata row
- `data-testid="cert-hash"` on the hash metadata row
- `data-testid="cert-key-id"` on the key id row
- `data-testid="cert-issued-at"` on the issued-at row
- `data-testid="cert-qr"` on the QR element
- `data-testid="cert-verify-footer"` on the footer text
- `data-testid="cert-download-manifest"` on the download button
- `data-testid="cert-share"` on the share button
- `data-testid="leak-terminal"` on the leak terminal container (future)

### Failure modes and timeouts
- **Backend cold start** shouldn't happen (always-warm), but first request gets a generous 30s timeout.
- **Navigation timeout**: `waitForURL` with `timeout: 30_000`.
- **Hash selector**: test uses `getByTestId`, not the truncated text, so the test is stable if the truncation format changes.
- **Flaky Monaco loading**: fall back to filling a plain textarea via `editor-root` — the `Editor` component guarantees both variants work.

## CI integration

### Blocking vs non-blocking
- S4 gate requires the happy-path spec to pass at least **locally** and optionally in CI.
- Adding Playwright to `ci.yml` as a **separate job** is a polish pass for S5/S6 (to avoid flaky CI blocking feature work early). For S4 we run it locally before committing.

### Optional CI job (S5/S6 polish)
```yaml
  e2e:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with: { version: 10 }
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: pnpm }
      - run: pnpm install --frozen-lockfile
      - run: pnpm playwright install --with-deps chromium
      - run: pnpm build
      - run: pnpm test:e2e
        env:
          NEXT_PUBLIC_API_URL: https://inkprint-backend.onrender.com
          NEXT_PUBLIC_SITE_URL: http://localhost:3000
          CI: "1"
```

The `playwright.config.ts` already shipped in S1 starts `pnpm dev` via `webServer`, so the above job will spin up the app, run the test, tear it down.
